# 数据持久化 PlayerPrefs


什么是数据持久化？
数据持久化就是将游戏运行时产生的数据存储到硬盘当中
即将内存的数据存储到硬盘当中进行保存
在下一次游戏运行时通过读取硬盘中的数据实现上一次游戏数据的继承
在这一过程中数据被持久保存在硬盘里
这就是数据持久化 描述数据的存取读取的两个过程
而 PlayerPrefs 就是实现数据持久化的方法之一

## GUI 基本控件

- GUI 控件绘制的共同点
	绘制这些 UI 就是去
	`OnGUI()` 这个类似生命周期函数的特殊函数
	里面调用对应 UI 控件的静态函数
	
	而这些函数的参数一般包含：
	- `Rect` 描述该 UI控件 位置 xy 坐标 和 wh 尺寸
	- `string` UI 中的显示文本
	- `Texture` 显示图片信息
	- `GUIContent` 综合信息
	- `GUIStyle` 自定义样式
	
	每一种 UI控件 都有多种重载
	无非就是这些参数的排列组合
	注意：排布 UI 的位置坐标原点(0,0) 为游戏窗口的左上角

## 游戏场景

- 场景切换时挂载 Mono 脚本的单例模式能否保证
	游戏开始运行的是开始场景
	单例的 instance 是在 Awake() 中进行赋值
	只要被挂载的对象不失活 单例就能成功赋值
	在游戏一开始就会为所有激活物体身上的脚本创建类对象并执行 Awake()
	而切换游戏场景会卸载开始场景 于是开始场景中创建的类对象全部销毁
	新的游戏场景重新创建当前场景中的脚本类对象 重新创建单例
	所以开始界面和游戏界面的 设置Panel 脚本对象完全不同
	每次切换场景产生的单例对象都是新创建的 先前的对象在切换场景时销毁了
	所以只要两个场景中挂载设置脚本的对象激活 单例持续存在
	说明挂载了 Mono 的脚本对象的单例的局限性是针对与当前加载的场景来说的
	当前加载的所有场景中有且仅允许一个该脚本类对象存在
	由于做了数据保存 所以哪怕销毁了原有单例对象 新的单例对象依旧能读取到保存的数据信息
	数据管理类没有继承 Mono 所以在第一次调用内部方法时被创建出来
	直至游戏结束才会被销毁
