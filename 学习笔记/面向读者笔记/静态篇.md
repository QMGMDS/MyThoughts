# 静态变量/方法/类怎么来的？

首先：
现在你作为程序在进行游戏开发，而你的策划告诉你他在设计一款游戏
想要有一个计分器，当怪物被玩家消灭时能增加对应分数，然后在各种游戏UI界面中显示
比如：游戏界面内实时显示、游戏结束界面显示最终分数
这对你来说并不难，你很快就写出了敌人类 `Enemy` 和分数管理类 `ScoreManager`
```C#
public class Enemy
{
    public int Score; // 该敌人的击杀得分
    
    public void Die()
    {
        // 如何增加玩家分数？
    }
}

public class ScoreManager
{
    public int playerScore = 0; // 记录玩家分数，初始为0
}
```
问题很快就出现了：
每次要访问 `playerScore` 的时候总不能去 `new` 吧，这些 `new` 出来的实例的 `playerScroe` 并不相通
而 `Enemy` 不知道当前游戏世界用的是哪个 `ScoreManager` 实例，难道要把分数管理实例传来传去吗？如果有114514个敌人，那么每个敌人死亡时都要先找到那个分数管理器实例，这样无疑复杂而又臃肿
该如何去找到那个专门记录分数的 `ScoreManager` 的实例，修改里面的分数呢？
于是你想到了：倘若有一个特殊的成员变量，让该类的所有实例都去共享它不就行了
紧接着......
**静态** `static` 应运而生~
它的核心思想是：**属于类型本身，而不属于任何单个对象**

于是你开开心心的写下了这段：
```C#
public class Enemy
{
    public int Score; // 该敌人的击杀得分
    
    public void Die()
    {
        ScoreManager.playerScore += Score;
    }
}

public class ScoreManager
{
    public static int playerScore = 0; // 记录玩家分数，初始为0
}
```
由于 `playerScore` 这个静态成员属于类型本身而非某个实例对象
所以直接使用 `类名.` 的方式进行使用，这样一来问题就迎刃而解了~
这样一来就结束了吧
使用**静态成员变量**来实现独所有类对象共享的变量

等等...你仔细琢磨了 静态`static` 这个关键字
类不仅仅封装了象征特征的成员变量，还有象征行为的成员方法
*华生，你发现了盲点！*
将 `static` 作为修饰放在成员方法身上，你发明了**静态成员方法**
像这样：
```C#
class MathTool
{
	// 取绝对值的方法
	public static int Abs(int x)
	{
	    return x > 0 ? x : (-x);
	}
}
```
现在对 `Abs()` 添加静态 `static` 修饰，你实现了
1. 跳过实例化对象而直接使用其内部方法
2. 保持对工具方法的封装性
在此前——
倘若我想要将所有关于数学计算的方法封装成一个工具类方便使用
就得先 `New` 出一个类对象，在通过类对象 `.` 点出来

但还不够，你发现了 `MathTool` 居然可以被 `new` 出来
而作为工具类，我只要使用内部方法就好了
也就是说，这些工具方法都是静态的啊，`new` 出来根本毫无意义
于是你将 `static` 放在了类 `class` 身上，**静态类**诞生了
```C#
static class MathTool
{
	public static float PI = 3.14f;
	// 取绝对值的方法
	public static int Abs(int x)
	{
	    return x > 0 ? x : (-x);
	}
}
```
静态类的特点就是：
1. 只能包含静态成员
2. 不能实例化

回过头来的你又完成了一项壮举
但你又疑惑了...

# 静态怎么实现的？

先来思考一下：静态的修饰改变了什么？
都说面向对象面向对象，将万物抽象为对象，为其封装各种成员变量和成员方法
按照面向对象的思想，想使用对象内部封装的内容，就得先创建对象
而这些没有被静态修饰的实例成员
它们的生命周期都是从对象的出生到对象的销毁
即：实例成员是依附于实例对象身上的

而先前被静态修饰的工具类内部方法的使用
是在该类对象出生前就被使用了
这不对吧，明明没有对象却能无中生有吗？

直到这里，敏锐的你发现了：
静态其实是提前对被修饰的变量/方法/类进行初始化
> **本质**：`static` 改变了成员的**归属**和**存活时间**，将其从“对象级”提升到“类型级”

在程序运行的最开始时，将这些静态的东西进行了**类型加载**，此时静态字段的槽位已分配，但值为默认值（`0`、`null`、`false`），此时还无法使用
而在首次访问类型时触发**类型初始化**，于是静态字段有了实际值，此时便可以使用
直到程序结束，这些存储在专门区域中的静态字段才会被释放

正是这样的机制，带来了：
- 静态方法中不能访问实例成员，而反之亦可，因为前者的生命周期的更早更长
- 大量静态成员会持续占用内存（因为存储在不受GC管理的Loader Heap中）

# 总结

静态成员属于类型本身而非某个实例对象
静态 `static` 实现的核心本质是：改变成员的生命周期

静态带来了 **单例模式** 和 **拓展方法** 的实现，有了类型级的数据表示，为后续描述 **类的信息类 `Type`** 的实现打下了坚实基础
