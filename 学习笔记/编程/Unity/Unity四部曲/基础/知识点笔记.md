# 3D数学

## 数学计算公共类 Mathf
- 概述
`Math` 是C#中封装好的用于数学计算的工具类 -> 位于 `System` 命名空间中
`Mathf` 是Unity中封装好的用于数学计算的工具结构体 -> 位于 `UnityEngine` 命名空间中
二者都是为了方便数学相关计算的
`Mathf` 是 Unity 专门封装的，不仅包含 `Math` 中的方法，还多了一些适用于游戏开发的方法
因此使用 `Mathf` 中的方法用于数学计算即可

- 计算多次的常用方法
`Lerp` 插值运算
函数公式：`result = Mathf.Lerp(start, end, t)`
其中 t 为插值系数，取值范围为 0~1，等价于下面的运算公式：
运算公式：`result = start + (end - start)*t`

1. 插值运算用法一
```C#
private float start = 0f; // 变

void Update()
{
	// 每帧改变start的值 -> 趋近速度先快后慢，位置无限接近，但是永远不会得到end位置
	start = Mathf.Lerp(start, 10, Time.deltaTime);
}
```

2. 插值运算用法二
```C#
private float start = 0f; // 不变
private float result = 0f;
private float time = 0f;

void Update()
{
	// 每帧改变t的值 -> 趋近速度匀速，位置每帧接近，当t>=1时，得到结果
	time += Time.deltaTime;
	result = Mathf.Lerp(start, 10, time);
}
```

- 向量点乘判断对象相对自身方位（前方和后方）
```C#
// 根据点乘结果判断目标物体在自身方位的前方还是后方
// 两个向量是否是单位向量对方位判断结果无影响
// this.transform.forward 即 自身方位的正前方
// target.position 即 目标物体的坐标
float dotResult = Vector3.Dot(this.transform.forward, target.position - this.transform.position);

if( dotResult >= 0 )
{  
    print("它在我前方");  
}  
else  
{  
    print("它在我后方");  
}
```
示意图：
左边物体 A 即自身，右边物体 B 为目标
$$
\vec{a} \cdot \vec{b} = |\vec{a}| \cdot |\vec{b}| \cdot \cos \theta
$$
其中 $|\vec{b}| \cdot \cos \theta$ 就是 b 向量在 a 向量方向上的投影向量的模长
点积的结果为 >0 则说明 B 在 A 的前方
点积的结果为 =0 则说明 B 在 A 正左侧或正右侧
点积的结果为 <0 则说明 B 在 A 的后方
就拿两向量点乘的结果本身没有几何意义，但通过结果正负判断有意义
![[Pasted image 20260406115605.png]]

- 向量点乘推导公式算出对象相对自身夹角（扇形）
```C#
// 用单位向量算出点乘结果 -> 夹角余弦值
float dotResult = Vector3.Dot(this.transform.forward, (target.position - this.transform.position).normalized);
// 用反三角函数得出弧度 随后转角度
print(Mathf.Acos(dotResult) * Mathf.Rad2Deg); // 夹角角度

// Vector3 中也提供了直接得到两个向量之间夹角的方法
// 传入要算夹角的两个向量，是否是单位向量对结果无影响
print(Vector3.Angle(this.transform.forward, target.position - this.transform.position)); // 夹角角度
```
$$
\cos \theta = \frac{\vec{a} \cdot \vec{b}}{|\vec{a}| \cdot |\vec{b}|}
$$
也可以看作两个方向的单位向量的点乘
夹角余弦值 -> 反余弦函数获得弧度 -> 弧度转角度 -> 两物体之间的夹角 (0~180°)
0~90° 代表 B 物体在 A 的前方
90~180° 代表 B 物体在 A 的后方 

- 向量叉乘判断对象相对自身方位（左侧和右侧）
```C#
// OA叉乘OB
// O为坐标原点
Vector3 C = Vector3.Cross(A.position, B.position);
if( C.y > 0)
{
    print("B在A的右侧");
}
else  
{
    print("B在A的左侧");
}
```
两向量叉乘的结果是得到一个垂直于两向量组成平面的新向量
对于 $\vec{a} \times \vec{b}$ ：
新向量的 y 值 > 0 -> 物体B在A的右侧
新向量的 y 值 < 0 -> 物体B在A的左侧
新向量的 y 值 = 0 -> 物体B在A的正前方或正后方
注意：叉乘不具有交换律
$$
\begin{align}
\vec{a} \times \vec{b} = \vec{C} \\
\vec{b} \times \vec{a} = \vec{D} \\
\vec{C} = -\vec{D}
\end{align}
$$

- Vector3 的线性插值 `Lerp`
```C#
public Transform A;
public Transform target;

private float time;
private Vector3 startPos;
private Vector3 nowTarget;

void Start()
{
	startPos = this.transform.position;
}

void Update()
{
	// 公式
    // result = start + (end - start) * t
    // Vector3.Lerp(start, end, t);
    
    // 先快后慢：每帧改变start位置 位置无限接近 但不会得到end位置  
    A.position = Vector3.Lerp(A.position, target.position, Time.deltaTime);
    
    // 匀速：每帧改变时间 当t>=1时得到结果
    if(nowTarget != target.position)
    {
	    nowTarget = target.position;
        time = 0;
        startPos = A.position;
    }
    time += Time.deltaTime;
    B.position = Vector3.Lerp(startPos, nowTarget, time);
}
```

## Quaternion 四元数
像欧拉角一样，四元数也是表示三维空间中物体旋转信息的一种方式

- 为什么使用四元数？
原因是使用欧拉角表示物体的旋转存在一些缺点：
1. 同一旋转结果的表示不唯一，即不同旋转路径能达到一样的最终效果
2. 万向节死锁
万向节死锁即：当某个特定轴达到某个特殊值时，绕一个轴旋转可能会覆盖住另一个轴的旋转从而失去一维自由度，比如 Unity 中 x 轴达到 90° 时会产生万向节死锁现象
而四元数旋转不存在万向节死锁问题
因此在计算机中往往使用四元数来表示三维空间中的旋转信息

- 概述
四元数的本质是描述三维空间中物体旋转信息的一种方法
通过 Unity 封装好的 `Quaternion` 结构体，可以解决欧拉角表示物体旋转信息时产生的问题

对于给定旋转，假设为绕着 $n$ 轴，旋转 $\beta$ 度，$n$ 轴向量表示为 $(x,y,z)$ 那么可以构成四元数为：
$$Q = (sin(\frac{\beta}{2})x, sin(\frac{\beta}{2})y, sin(\frac{\beta}{2})z, cos(\frac{\beta}{2}))$$
最后一个表示 $w$ 旋转量，是个标量，决定了旋转的角度

- 四元数的声明
```C#
// 绕轴 (1,0,0) 旋转60°
// (1,0,0) -> (x,y,z)
// 60° -> β
// Quaternion q = new Quaternion(x, y, z, w);
Quaternion q = new Quaternion(Mathf.Sin(30 * Mathf.Deg2Rad), 0, 0, Mathf.Cos(30 * Mathf.Deg2Rad));
```
一般都不会通过这种方式声明，而是直接调用封装好的 API
```C#
// 绕轴 (1,0,0) 旋转60°
Quaternion q = Quaternion.AngleAxis(60, Vector3.right);
```

- 四元数和欧拉角的转换
```C#
// 欧拉角转四元数
Quaternion q = Quaternion.Euler(60, 0, 0);
// 四元数转欧拉角

```

- 单位四元数

- 四元数的线性插值 `Lerp`

## 延迟函数
继承 Mono 的脚本才能使用延迟函数
延迟函数指会延迟执行的函数，是 Mono 中写好的函数方法

- 延迟函数的使用
注意：填入的函数名只能是在使用延迟函数的类中，并且没办法直接调用需要参数的函数，如果要让外部类的方法延迟执行或者调用带参数的函数只能通过包裹一层函数来解决

只会执行一次的延迟函数
```C#
// 参数一：要延迟执行的函数名 填入字符串
// 参数二：要延迟的时间 单位为秒
Invoke("DelayDoSomething", 1);

private void DelayDoSomething()  
{  
    print("延时执行的函数");
}
```

会重复执行的延迟函数
```C#
// 参数一：要延迟执行的函数名 填入字符串
// 参数二：第一次执行的延迟时间
// 参数三：之后每次执行的间隔时间
InvokeRepeating("DelayRe", 5, 1);
```

取消全部延迟函数
```C#
// 取消该脚本上的所有延时函数执行
CancelInvoke();
```

取消指定的延迟函数
```C#
// 指定函数名取消
// 注意：
// 只要取消了指定函数的延迟执行
// 不管之前该函数先前开启了多少次延迟执行 都会被统一取消
CancelInvoke("DelayDoSomething");

private void DelayDoSomething()  
{  
	print("取消该函数的延迟执行");
}
```

延迟函数的存在判断
```C#
// 是否有任意的延迟函数存在且没被取消
if( IsInvoking() )  
{  
    print("存在延迟函数可以被执行");  
}
// 是否有指定的延迟函数存在且没被取消
if( IsInvoking("DelayDoSomething") )  
{  
    print("存在延迟函数 DelayDoSomething 可以被执行");  
}
```

- 延迟函数执行与对象激活失活的关系
有延迟函数的脚本，其依附对象失活或脚本自身失活不会导致延迟函数的执行被暂停
只有脚本依附对象销毁或脚本移除或利用取消延迟函数代码执行才能让延迟函数无法继续执行

## 协同程序

- Unity 是否支持多线程？
支持，但新开的线程无法访问 Unity 相关对象的内容，并且新线程如果不关闭的话会默认和编辑器一样一直执行，直到关闭 Unity 编辑器，可以在新线程中专门处理寻路网络相关复杂的计算，将返回结果丢给公共的容器之中供主线程取使用

- 协同程序（协程）是什么？
**协程不是多线程**
协程是在原线程上开启的，主要作用可以让代码分时分步执行
不卡主线程，用时间换流畅
具有这样特点的程序执行逻辑被称为协程，一般通过协程函数实现协程

- 协程本质
协程的实现可以分成两部分：
1. 协程函数本体（实现分布执行）
2. 协程调度器（实现定时执行）

协程本体就是一个 C# 的迭代器方法，实现了一个能够中间暂停返回的函数
协程调度器是 Unity 内部实现的，会在对应的时机继续执行协程函数

Unity只实现了协程调度部分
协程的本体本质上就是一个 C# 的迭代器方法

协程就是利用
C# 的迭代器函数"分步执行"的特点与协程调度逻辑
实现的一套分时执行函数的规则

关于手动实现 Unity 中的简易协程调度器（`StartCoroutine`）
以及更深入的了解，请看这篇 [[协程篇]]

- 协程函数的声明
```C#
// 协程函数的返回值必须是 IEnumerator
IEnumerator MyCoroutine(int i, string str)  
{
    print(i);
    // 协程函数当中 必须使用 yield return 进行返回
    // yield return 将函数的执行过程变成上下两部分
    // null 代表执行等待的间隔时间，这里是 起码等待一帧
    // 主线程将 print(i) 执行完毕后不会执行 print(str)
    // 而是继续往下走，直到完成一次主线程的循环（完成一帧）
    // 而后继续执行 print(str)
    yield return null;
    print(str);
}
```

- 协程函数的调用（协程的开启）
```C#
void Start()
{
	// 像调用函数一样，但一定要 StartCoroutine() 来调用
	StartCoroutine( MyCoroutine(1, "123") );
	// StartCoroutine() 方法的返回值是 Coroutine
	Coroutine c1 = StartCoroutine( MyCoroutine(1, "123") );
}
```

- 协程函数的关闭（协程的关闭）
```C#
// 关闭所有协程
StopAllCoroutines();
// 关闭指定的协程
StopCoroutine("MyCoroutine"); // 通过函数名字关闭
StopCoroutine(c1); // 通过执行协程后的 Coroutine 参数关闭
```

- `yield return` 的不同含义
```C#
// 1.下一帧执行 后面逻辑
// 数字代表了下几帧后执行
// null 则表示下一帧后执行
yield return 数字;
yield return null;
// 执行的时机在 Update 和 LateUpdate 之间

// 2.等待指定秒后执行 后面逻辑
yield return new WaitForSeconds(秒);
// 执行的时机在 Update 和 LateUpdate 之间

// 3.等待下一个固定物理帧更新后执行 后面逻辑
yield return new WaitForFixedUpdate();
// 执行的时机在 FixedUpdate 和 碰撞检测相关函数 之后

// 4.等待摄像机和GUI渲染完成后执行 后面逻辑
yield return new WaitForEndOfFrame();
// 执行的时机在 LateUpdate 和 渲染相关处理完毕 之后

// 5.一些特殊类型的对象 比如异步加载相关函数返回的对象
// 执行的时机在 Update 和 LateUpdate 之间

// 6.跳出协程
yield break;
```

## 特殊文件夹
指 Unity 中有专门方法获取路径和频繁使用文件夹
- 获取工程路径
该路径仅在编辑器下使用，当游戏被打包时该路径就不存在了
```C#
// 打印出工程 Assets 文件的绝对路径，例如：
// E:/Unity Project/Unity_TangTeach2/Assets
print(Application.dataPath);
```

- `Resources` **资源文件夹**
该文件夹需要自行在 `Assets` 文件夹中创建名为 `Resources` 的文件夹
没有专门的路径获取的 API
作为资源文件夹，其特点为：
1. 需要通过 `Resources` 相关 API 动态加载的资源需要放在其中
2. 该文件夹下所有文件都会被打包出去
3. 打包时 Unity 会对其压缩加密
4. 该文件夹打包后内容为只读，只能通过 `Resources` 相关 API 加载

- `StreamingAssets` **流动资源文件夹**
该文件夹需要自行在 `Assets` 文件夹中创建名为 `StreamingAssets` 的文件夹
路径获取：
```C#
// 不同平台中会自行打印出不同的路径，例如编辑模式下路径：
// E:/Unity Project/Unity_TangTeach2/Assets/StreamingAssets
print(Application.streamingAssetsPath):
```
为流动资源文件夹，其特点为：
1. 打包出去不会被压缩加密
2. 移动平台只读，PC平台可读可写
3. 一般放入一些需要自定义动态加载的初始资源

- `persistentDataPath` **持久数据文件夹**
该文件夹不需要自行创建，该路径在不同平台中会自行打印出不同的路径
路径获取：
```C#
// 不同平台中会自行打印出不同的路径，例如 Windows 下路径：
// C:/Users/29182/AppData/LocalLow/DefaultCompany/Unity_TangTeach2
print(Application.persistentDataPath);
```
为持久数据文件夹，其特点为：
1. 所有平台都可读可写
2. 一般用于放置动态下载或者动态创建的文件，游戏中创建或者获取的文件都放在其中

- `Standard Assets` 默认资源文件夹
该文件夹需要自行在 `Assets` 文件夹中创建名为 `Standard Assets` 的文件夹
一般不获取路径
为默认资源文件夹，其特点为：
1. 一般Unity自带资源都放在这个文件夹下，代码和资源优先被编译

- `Plugins` 插件文件夹
该文件夹需要自行在 `Assets` 文件夹中创建名为 `Plugins` 的文件夹
一般不获取路径
为插件文件夹，其特点为：
1. 不同平台的插件相关文件放在其中

- `Editor` 编辑器文件夹
该文件夹需要自行在 `Assets` 文件夹中创建名为 `Editor` 的文件夹
一般不获取路径
为默认资源文件夹，其特点为：
1. 开发Unity编辑器时，编辑器相关脚本放在该文件夹中
2. 该文件夹中内容不会被打包出去

## Resources 资源同步加载
通过代码动态加载 `Resources` 文件夹下指定路径的资源
比如：
1. 预设体对象 -> GameObject
2. 音效文件 -> AudioClip
3. 文本文件 -> TextAsset
4. 图片文件 -> Texture
注意：
预设体对象加载后需要实例化，其它资源加载一般直接用
重复加载相同的资源不会浪费内存（Unity会判断当前加载资源是否已经被加载过），但会浪费性能（每次都需要判断资源是否被加载过）
`Resources.Load` 会加载不同路径下的 `Resources` 文件夹中资源，在打包时将这些资源统一放在一个文件夹中
而加载同名资源时，会无法准确加载出想要的内容，可以使用另外的 API 指定加载

- 正常资源同步加载 API
```C#
// #加载预制体资源
// 第一步：要去加载预设体的资源文件(加载配置数据在内存中)
Object obj = Resources.Load("Cube");
// 第二步：如果想要在场景上创建预设体 一定是加载配置文件过后再实例化
Instantiate(obj);

// #加载音效资源
// 第一步：加载音效的资源文件
Object obj3 = Resources.Load("Music/BKMusic");
// 第二步：使用时不需要实例化音效切片 只需要把数据赋值到正确的脚本上即可
public AudioClip audioS; // 外部拖曳赋值
audioS.clip = obj3 as AudioClip;
audioS.Play();

// #加载文本资源
// 第一步：加载音效的资源文件
TextAsset ta = Resources.Load("Txt/Test") as TextAsset;
// 第二步：打印文本内容
print(ta.text);
print(ta.bytes); // 打印字节数据组

// #加载图片资源
// 加载图片的资源文件
Texture tex = Resources.Load("Tex/TestJPG") as Texture;
```

- 指定类型资源同步加载 API
```C#
// 以加载图片资源为例

// #加载指定类型的资源
Texture tex = Resources.Load("Tex/TestJPG", typeof(Texture)) as Texture;

// #加载指定名字的所有资源
Object[] objs = Resources.LoadAll("Tex/TestJPG");

// #资源同步加载的泛型方法（一般都使用这种方法）
Texture tex = Resources.Load<Texture>("Tex/TestJPG");
```

## Resources 资源异步加载
与 `Resources` 同步加载相比，`Resources` 异步加载，即内部新开一个线程进行资源加载而不会造成主线程卡顿
相对于同步加载的缺点就是不能直接拿来用，起码得等一帧的时间后才能加载完毕

- 异步加载资源和使用的两种方式
1. 通过异步加载中的完成事件的监听来使用加载的资源
```C#
// 存储加载完毕后的资源
private Texture tex;

void start()
{
	// Unity 开了一个新线程专门处理资源加载
	// 当资源加载完成后会自动调用内部名为 completed 的委托
	ResourceRequest rq = Resources.LoadAsync<Texture>("Tex/TestJPG");
	// 为委托添加函数，当资源被加载完成后自动执行
	rq.completed += LoadOver;
}

// 函数的参数和返回值要与委托一致
private void LoadOver(AsyncOperation rq)
{
    print("加载结束");
    // asset 即是资源对象
    // 加载完毕过后通过使用它来使用资源
    tex = (rq as ResourceRequest).asset as Texture;
    // 加载完毕后直接使用 tex
}
```

2. 通过协程来使用加载的资源
```C#
// 存储加载完毕后的资源
private Texture tex;

void start()
{
	StartCoroutine(Load());
}

IEnumerator Load()
{
	ResourceRequest rq = Resources.LoadAsync<Texture>("Tex/TestJPG");
	// Unity 会自己判断该资源是否加载完毕
	// 加载完毕后 才会继续执行后面的代码
	yield return rq;
	// 资源加载完毕了
	tex = rq.asset as Texture;
	// 加载完毕后直接使用
}
```
通过 `ResourceRequest` 的父类 `AsyncOperation` 中的属性来获取当前资源加载的进度
```C#
IEnumerator Load()
{
	ResourceRequest rq = Resources.LoadAsync<Texture>("Tex/TestJPG");
	// 判断资源是否加载结束
	while(!rq.isDone)
	{
	    // 打印当前的 加载进度
		// 该进度不会特别准确 过渡也不是特别明显
	    print(rq.priority);
	    yield return null;
	}
	// 资源加载完毕了
	tex = rq.asset as Texture;
	// 加载完毕后直接使用
}
```

## Resources 手动卸载资源

- 卸载指定资源
注意：该方法不能释放 GameObject 对象，因为它会用于实例化对象，即使是没有实例化的 GameObject 对象也不能进行卸载
只能用于一些不需要实例化的内容，比如图片、音效和文本等
```C#
// 加载资源
Texture tex = Resources.Load<Texture>("TextJPG");
// 卸载资源
Resources.UnloadAsset(tex);
```

- 卸载未使用的资源
一般在过场景时和GC一起使用
```C#
// 卸载未使用的资源
Resources.UnloadUnusedAssets();
// 触发 GC 回收
GC.Collect();
```

# 场景异步切换

场景异步加载和资源异步加载几乎一致，有两种方式
1. 通过事件回调函数异步加载
```C#
void Start()
{
	// 异步加载场景
	AsyncOperation ao = SceneManager.LoadSceneAsync("TestScene");
	// 为委托添加函数，当资源被加载完成后自动执行
	ao.completed += LoadOver;
	// 或者简写
	ao.completed += (a) =>
	{
	    print("加载结束");
	};
}
// 函数的参数和返回值要与委托一致
private void LoadOver(AsyncOperation rq)
{
	print("场景加载完毕");
}
```

2. 通过协程异步加载
```C#
void Start()
{
	StartCoroutine(LoadScene("TestScene"));
}

IEnumerator LoadScene(string name)
{
	// 异步加载场景
	AsyncOperation ao = SceneManager.LoadSceneAsync(name);
	// 在这之前就可以写一些别的逻辑 比如进度条加载
	// 等待场景加载完成
	yield return ao;
	// 在这里场景就会加载完成，原脚本依附对象被移除
	// 就会导致 yield return 后面的逻辑不被执行
	// 解决办法就是使用 DontDestroyOnLoad(this.gameObject)
	// 来让被挂载对象不被卸载
}
```

### 最初

- 范围检测的必备条件：
1. 想要被范围检测的对象必须具备碰撞器

- 范围检测的特点：
1. 范围检测相关API 只有当执行该句代码时进行一次范围检测 它是瞬时的
2. 范围检测相关API 并不会真正产生一个碰撞器 只是碰撞判断计算而已

- 范围检测相关 API
这些 API 的重载无非就是参数的数量不同，知道最多数量的参数就可以知道所有 API 的功能

1. API 共有参数介绍
**检测指定层级参数**
不填默认检测所有层
层级指 Layer，Unity 中最多支持设置 0~31 共计 32 个层级
这里填入参数类型的是 int 即层级代表的数字（层级索引）
Unity 中对所有代表层级的数字做出了处理：
第 0 层即 1 左移 0 位
第 1 层即 1 左移 1 位
第 2 层即 1 左移 2 位
......
第 0 层级 -> 1    -> 0000 0000 0000 0000 0000 0000 0000 0001
第 1 层级 -> 2    -> 0000 0000 0000 0000 0000 0000 0000 0010
第 2 层级 -> 4    -> 0000 0000 0000 0000 0000 0000 0000 0100
第 3 层级 -> 8    -> 0000 0000 0000 0000 0000 0000 0000 1000
第 4 层级 -> 16  -> 0000 0000 0000 0000 0000 0000 0001 0000
第 5 层级 -> 32  -> 0000 0000 0000 0000 0000 0000 0010 0000
第 6 层级 -> 64  -> 0000 0000 0000 0000 0000 0000 0100 0000
......
刚好对应了 int 作为 32 位二进制数时，1 在每一个位置上的情况
如此一来，如果我想要同时范围检测第2和第4层的物体
只需要填入两个层级的二进制数表示之和的数字 16+4=20 即可
结果就是一个数字就可以表示多个层级之间的组合判断
比如说我现在想要范围检测在第2和第4层级中的物体：
则该检测指定层级参数填入20，或者填入：
`1 << LayerMask.NameToLayer("Default") | 1 << LayerMask.NameToLayer("UI")`
假如第2和第4层级分别为 Default 和 UI 层
两种写法的效果一样，因为计算的结果都是 20

**是否忽略触发器参数**
不填默认使用 `UseGlobal`
全局设置指的是 `ProjectSettings/Physics/Queries Hit Triggers` 中是否勾选，是否对触发器也进行范围检测
`UseGlobal` -> 使用全局设置
`Collide` -> 检测触发器
`Ignore` -> 忽略触发器

## 射线检测

- 射线检测的必备条件：
1. 想要被射线检测的对象必须具备碰撞器

- 射线检测的特点：
1. 射线检测相关API 只有当执行该句代码时进行一次射线检测 它是瞬时的
2. 射线检测相关API 并不会真正产生一个碰撞器 只是碰撞判断计算而已

- 射线对象
单独的射线没有实际的意义，需要配合物理系统进行射线碰撞判断

```C#
// 代表一条射线的位置和方向信息

// 参数二：射线射出方向 direction
Ray r1 = new Ray(Vector3.right, Vector3.forward);

// 摄像机发出的射线
// 一条从屏幕位置作为起点
// 摄像机视口方向为方向的射线
Ray r2 = Camera.main.ScreenPointToRay(Input.mousePosition);
```

- 射线碰撞信息类
存储射线碰撞到的物体身上的各种信息
```C#
RaycastHit hitInfo = new RaycastHit();

// 碰撞器信息
print(hitInfo.collider.gameObject.name);  
// 碰撞到的点（对象身上表面的点）
print(hitInfo.point);
// 法线信息
```
